# 003 — Fixtures: TS catalog files instead of CMS for Phase 1

Status: accepted (2026-04-29)

## Context
Vision Affichage's Phase 1 catalog is small and slow-changing: ~21 SKUs, 6 industries, 8 client logos, 8 reviews. An external CMS adds latency, vendor lock-in, and an extra deploy surface for a 2-person operator team.

## Decision
All catalog data lives in typed TypeScript fixtures under `lib/`:
- `lib/products.ts` — Product[] keyed by `styleCode`
- `lib/industries.ts` — Industry[] keyed by `slug`
- `lib/clients.ts` — ClientLogo[]
- `lib/reviews.ts` — Review[] + helper aggregations
- `lib/site.ts` — SiteConfig (org name, addresses, social URLs)

`lib/types.ts` defines the shared shapes; `tsc --strict` plus `noUncheckedIndexedAccess` keeps every consumer honest.

## Rationale
- Zero runtime fetches; pages render at build time via `generateStaticParams`.
- Editing copy is a code review, with diff history and rollback.
- Bilingual literal types (`Bilingual<T>`) catch missing fr-CA/en-CA pairs at compile time.
- No CMS auth, schema, or dashboard to keep alive.

## Consequences
- Non-developer edits require a PR. Acceptable for the operator's current cadence.
- Image assets live in `/public/placeholders/` and are SVG-generated by `scripts/generate-placeholders.cjs`. Real photography replaces these in Phase 2 (see operator queue).

## Migration path to a headless CMS
When the catalog crosses ~100 SKUs or non-developer editing becomes a bottleneck:
1. Pick a headless CMS (Sanity / Contentful / Storyblok). Map fixture types 1:1.
2. Replace each `lib/*.ts` consumer with a typed fetch wrapper (Server Component, cached via Next.js `fetch` revalidate).
3. Keep the `Bilingual<T>` shapes — the CMS schema mirrors them.
4. Add a webhook on publish that re-revalidates the affected static routes via `revalidatePath`.
